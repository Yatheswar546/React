useEffect: used to handle side-effects in React application.
=> In the above example "fetchAPI" which is outside of React is making the website slow by infinite loop.
=> setTimeouts, LocalStorages, Read/Write files etc...

Syntax: useEffect((callback) => { Dependency Array})

Eg: without Dependency list
    
    useEffect(() => {
        console.log(products, "products");
        fetchData();
    });

Eg: with Dependency list

    useEffect(() => {
        console.log(products, "products");
        fetchData();
    }, []); //mounting

    In this fetchData() renders only once, because of Dependency List 
    This is also known as "Mounting Phase".

Solution: Promise Concurrency
            |-> Promise.all()
            |-> Promise.allSettled()
            |-> Promise.any()
            |-> Promise.race()

let consider timetaken for 3 tasks 
1 ==> 3 secs
2 ==> 5 secs
3 ==> 4 secs 

In normal execution : 3 + 5 + 4 = 12 secs

In Promise Concurrency:

after 3 secs 
1 ==> done 
2 ==> 2 secs 
3 ==> 1 sec 

after 4 secs 
1 ==> done 
2 ==> 2 secs 
3 ==> done

after 5 secs 
1 ==> done 
2 ==> done 
3 ==> done

let postApi = await fetch("https://........./posts");
let userApi = await fetch("https://........./users");
let commentsApi = await fetch("https://........./comments");
    
async function fetchData(){
    let startTime = new Date();
    
    let [postRes, userRes, commentRes] = await Promise.all([
        fetch(postApi),
        fetch(userApi),
        fetch(commentsApi),
    ]);

    let [posts, users, comments] = await Promise.all([
        postRes.json(), 
        userRes.json(), 
        commentRes.json(),
    ]);

    console.log(posts, users, comments, "res");

    let endTime = new Date();
    let timeTaken = endTime - startTime;
    console.log(timeTaken);
}

export { fetchData };

Debouncing:
=> It is a programming practice that ensures a func. is not called too frequently 
    by delaying its execution until a specified time has passed since the last invocation.
=> Use Cases: Search Autocomplete, Window Resizing, Button clicks etc...

Problem Statement: 
In general in search input fields whenever we type a single letter the API call is done multiple times.
Because of this API is called multiple times increasing in bandwidth and decreasing the performance.

Solution: (Debouncing)
=> Debouncing is used to make a particular function to run after a specific time reducing the load.

Later apply Debouncing and observe the change.

function Header(){
    const [searchKey, setSearchKey] = useState("");
    const [timer, setTimer] = useState("");

    const handleChange = (e) => {
        setSearchKey(e.target.value);
    };

    const fetchVideos = async () => {
        try{
            let res = await axios.get(
                `https://................videos?title=${searchKey}`,
            );
            console.log(res.data);
        } catch (err) {
            console.log(err, "error");
        }
    };

    function debounceSearch(func, delay){
        if(timer) clearTimeout(timer);
        let timerId = setTimeout(() => {
            fetchVideos()
        }, delay);
        setTimer(timerId);
    }

    useEffect(() => {
        debounceSearch(1000);
    }, [searchKey]);   // calls the function only when searchKey is updated. known as "update phase in useEffect".

    return(
        <TextField
            type="text"
            placeholder="Search here..."
            value={searchKey}
            onChange={(e) => handleChange(e)}
        />
    )
}

Lifting State Up:

Problem Statement:
The videos we are searching in search bar in Header.js should be updated in videos section in Videos.js
But videos and header are not parent-child so props won't work in this case.
So, how to achieve this ?

Solution:
Header and Videos are siblings to each other in App.js
We can share the data to common parent (App.js) and then share the data to children (Header & Videos).
This is called as "Lifting the State Up".

Example Code:
App.js:

function App(){

    const [searchKey, setSearchKey] = useState("");
    const [timer, setTimer] = useState("");
    const [videos, setVideos] = useState([]);

    const url = searchKey
        ? `https://................videos?title=${searchKey}` : 
        : `https://................videos`;

    const fetchVideos = async () => {
        try{
            let res = await axios.get(url);
            console.log(res.data);
            setVideos(res.data.videos);
        } catch (err) {
            console.log(err, "error");
        }
    };

    function debounceSearch(func, delay){
        if(timer) clearTimeout(timer);
        let timerId = setTimeout(() => {
            fetchVideos()
        }, delay);
        setTimer(timerId);
    }

    useEffect(() => {
        debounceSearch(1000);
    }, [searchKey]); 

    return(
        <>
            <Header searchKey={searchKey} setSearchKey={setSearchKey} />
            <Videos videos={videos}/>
        </>
    )

}

"bind" method:
=> When we pass the method as callback, the value of this is lost and when the executor function executes the callback, 
    this is not pointing to the student object anymore.

    Eg: const student = {
            fullName: "Rohit",
            greet: function(){
                console.log("Hello, " + this.fullName);
            },
        };

        function executeObjectFunction(callback){
            callback();
        }

        executeObjectFunction(student.greet);

        // The above code's output is : Hello, undefined.

=> what we can do is pass the value of "this" explicitly to the "student.greet" method.
=> To achieve this we can use "bind" method which can be applied on all methods.

    Eg: const greet = student.greet.bind(student); // bind the object to thee method
        executeObjectFunction(greet); // Passing the object method as callback

=> The bind method returns a new function.
=> It helps us set the value of this to the passed value for specific function.

    Eg: const student = {
            fullName: "Rohit",
            greet: function(){
                console.log("Hello, " + this.fullName);
            },
        };

        function executeObjectFunction(callback){
            callback();
        }

        const greet = student.greet.bind(student)
        executeObjectFunction(greet);

=> FP is the process of building software by composing pure functions, 
    avoiding shared state, mutable (changing) data and side-effects.
=> FP is declarative (what needs to be done) rather than

    }

=> Functional code tends to be more concise, more predictable, and easier to test

Pure Functions:
=> A function which given the same inputs, always returns the same output (immutability).
=> Has no "side-effects".
=> These are functions which returns a predictable and deterministic value.

Eg:
// Pure function
function addTwo(a,b){
    return a+b;
}

// Impure function

    console.log(a,b);
}

Advantages:  
=> Works as an independent function that gives same output for same inputs.
=> Readable because of independent behavior.
=> Straightforward to debug.
=> Can clone an external state into a pure function, but it doesn't change the purity of function.

=============================================================================

Side-Effects:
=> A side effect can be perceived as something a function does, which does not pertain to returning a value.
=> Examples - printing a value, using global variables, doing external operations like changing the values of a different function, 
    or even mutating the values/objects received as arguments
=> A function is pure when it does not have side effects.

Examples:
=> Modifying any external variable or object property (e.g., a global variable, or a variable in the parent function scope chain)
=> Logging to the console
=> Writing to a file
=> Calling any other functions with side-effects

Immutability:
=> An Immutable object is that which can’t be modified after it’s created. 
=> Conversely, a mutable object is any object which can be modified after it’s created.
=> Immutability of a pure function means that the function would treat its input values as immutable (once created, cannot be changed)
=> So if we wish to perform a mutation on the input, we first copy it and then mutate its value.

Eg: An array.push(value) will mutate the original array with the value passed in the push function.

function purePush(arr, value) {

  nextArr.push(value);
  return nextArr; //returing the new state
}

function purePush(arr, value) {
  return [...arr, value]; //returns new array with elements from arr and value.

        "NO UNNECESSARILY RE-RENDERS"
=> If there are more re-renders in a short-span of time then the browser will run slow or get crashed.
=> A setter method always re-render a component, below is a example.

Eg: App.js
import { useState } from "react";

export default function App(){
    // 1st approach: Use simple Js variable
    let counter = 0;
    // 2nd approach: Use useState variable
    const [count, setCount] = useState(0);

    const handleIncrement = () => {
        setCount((prevValue) => prevValue + 1);
        counter++;
        console.log("count: ", counter);
    };

    console.log("Component re-rendered");

    return(
        <div className="App">
            <h2>{count}</h2>
            <button onClick={handleIncrement}>Increment</button>
        </div>
    );
}

=> To solve the above re-rendering problem the below approaches won't work effectively.
    1st approach: Use simple Js variable
    2nd approach: Use useState variable
=> To solve this issue we use "useRef" hook. (3rd approach)

useRef:
=> "useRef" is used to create mutable reference persisting across multiple re-render for an element while avoiding unnecessary re-renders.
=> The useRef will updates the component(data variable) without re-rendering.
=> The value will be persistent across multiple re-renders which was not possible with 'useState'.

Demo code: App.js
import { useState } from "react";

export default function App(){

    // 3rd approach: useRef
    const counterRef = useRef(0);

    const handleIncrement = () => {
        counter++;
        console.log("count: ", counter);

        // writing a useRef variable
        counterRef.current++;
        console.log("useRef variable value", counterRef.current);  // reading a useRef variable
    };

    console.log ("Component re-rendered");

    return(
        <div className="App">
            <h2>{count}</h2>
            <button onClick={handleIncrement}>Increment</button>
        </div>
    );
}

NOTE: the reading and writing of a useRef variable can be done only at two places.
        1. handler functions 
        2. useEffect hooks 

Pitfalls of useRef Hook:
=> Do not read/write ref.current during rendering.
=> React expects that components should behave like a pure function.
=> If the inputs(props, state and context) are the same, it should return exactly the same JSX.

Use cases of useRef Hook:
=> DOM Manipulation
=> Focusing a text input

Eg:

useMemo:

=> Expensive function: A function that is executed multiple times but gives same input increasing bandwidth.
Eg: 
function square(num){
    console.log('Square ', num);
    return num * num;
}

=> To avoid expensive function we use "caching function" that helps us avoid operations that gives same values for same inputs.

Eg:
function memoizeFunction(fn){
    const cache = {};

    return function (input){
        if(cache[input]){
            console.log('Fetching from cache...');
            return cache[input];
        } else {
            const result = fn(input);
            cache[input] = result;
            return result;
        }
    };
}

const memoizedFn = memoizeFunction(square);

In the above code, memoizeFunction function stores the computed value, 
and if the input is already calculated, it gives the result instead of re-calculating it.

Disadvantages:
=> You have to declare a new variable to store input and results.
=> You have to check if the value is already calculated.
=> You have to assign values and store once they’re calculated.

Solution:
useMemo can help us achieve the same "memoization" with less lines of code in an optimized manner.

3 types of Memoization:
1. Memoization of values.
2. Memoization of function handlers.
3. Memoization of entire component.

Example:
import React, { useState, useMemo } from "react";

function SquareComponent({ input }) {
    function square(number) {
        console.log("Square ", number);
        return number * number;
    }
}

const memoizedSquare = useMemo(() => square(input), [input]);

return (
    <div>
        <h1>Memoized Square Function</h1>
        <p>Result: {memoizedSquare}</p>
    </div>
);

function MyComponent() {
    const [render, setRender] = useState(false);
    const [number, setNumber] = useState(1);

    const handleIncrement = () => setNumber(number + 1);

    console.log("Component Re-rendered");

    return (
        <>
            <SquareComponent input={number} />
            <button onClick={() => setRender(!render)}>Re-Render</button>
            <button onClick={handleIncrement}>Increment Number</button>
        </>
    );
}

export default MyComponent;

Uses of "useMemo":
=> By using useMemo, React automatically handles the caching for us.
=> It ensures that computation is only performed when the input prop changes.
=> Avoiding unnecessary computations on subsequent renders with the same input value.
=> useMemo actually returns calculated values and memoize that value.
